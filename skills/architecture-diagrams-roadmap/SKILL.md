# Architecture Roadmap — Strategy Execution Timeline

An Architecture Roadmap is a visual, time-based representation of strategic
work packages and initiatives. It plots the execution of strategy against a
timeline and links each activity to the goals and objectives it supports.

This skill is for architects who need a clear, visual schedule of how strategy
will be implemented and monitored over time.

---

## When to use this skill

Use this skill when you need to:

- Convert the Work Package Portfolio Map into a timeline view.
- Show when initiatives will start and finish.
- Link initiatives to strategic goals and objectives.
- Track ownership, status, and cost of execution through time.
- Provide stakeholders with a forecast of change and delivery.

---

## Outcome

Produce an Architecture Roadmap that shows:

- **Goals and Objectives**: listed vertically on the left.
- **Timeline**: plotted horizontally, typically by years and quarters.
- **Work Packages / Initiatives**: displayed as bars or blocks across the timeline.
- **Additional properties**: owner, status, cost, and progress coloring.

The roadmap should make it easy to answer when each initiative will occur, who
is accountable, how it supports strategy, and whether it is on track.

---

## Step-by-step workflow

1. **Source the work package data**
   - Start with the Work Package Portfolio Map.
   - Include goals, objectives, initiatives, owners, and relevant metadata.

2. **Prepare timeline details**
   - Determine the planning horizon (e.g., multi-year, quarterly view).
   - Capture start and end dates or estimated timeframes for each initiative.

3. **Build the roadmap layout**
   - Place goals and objectives vertically along the left side.
   - Create a timeline axis across the top or bottom.
   - Plot each work package as a horizontal bar spanning its scheduled dates.

4. **Enrich with attributes**
   - Add owners, progress state, and estimated cost information to each bar.
   - Use color coding to indicate status such as not started, in progress,
     completed, or canceled.
   - Optionally annotate bars with business units, priorities, or risk levels.

5. **Review for coverage and feasibility**
   - Verify that each initiative clearly maps back to a goal or objective.
   - Confirm timeline realism with delivery owners and sponsors.
   - Check for resource conflicts and execution risk.

6. **Validate and communicate**
   - Present the roadmap to stakeholders for approval.
   - Use it as a governance tool to monitor progress, costs, and changes.
   - Keep the roadmap updated as timelines, statuses, or priorities shift.

---

## Decision points and guidance

- **Timeline granularity**: choose a level of detail that fits the planning horizon;
  use quarters for multi-year roadmaps and months for shorter execution windows.
- **Visual clarity**: avoid overcrowding by limiting the number of initiatives shown
  in a single view or by grouping related work packages.
- **Color view**: use status-based coloring to provide an at-a-glance view of
  initiative progress.
- **Roadmap purpose**: treat the roadmap as a communication and monitoring tool,
  not just a static schedule.
- **Ownership visibility**: display the responsible business unit or role to make
  accountability clear.

---

## Quality criteria

A strong Architecture Roadmap should be:

- **Timely**: shows when each initiative is planned and how long it will run.
- **Strategically aligned**: each work package is tied to a goal or objective.
- **Informative**: includes owner, status, and cost or progress indicators.
- **Readable**: uses a clean timeline layout without excessive visual clutter.
- **Living**: maintained as the execution plan evolves and changes.
