# Minimum Viable Solution

This skill helps AI explain how to use minimum viable solutions in project
planning—focusing teams on the smallest outcome-driven release that still
matters, while recognizing uncertainty and the need to learn.

---

## When to use this skill

Use this skill when you need to:

- help teams choose the smallest release that achieves a valuable outcome,
- distinguish good MVP thinking from bad, overly minimal product decisions,
- move planning from feature delivery to outcome delivery,
- surface assumptions, risks, and the need for learning,
- align release slices with real-world business events and deadlines.

---

## Outcome

Produce guidance that:

- defines minimum viable solution as the smallest release that achieves desired outcomes,
- avoids the “crappiest product you could possibly release” trap,
- helps teams focus on users, goals, and behavior changes rather than features,
- encourages outcome-based slicing of story maps for release planning,
- frames the first release as a hypothesis to be validated, not a final product.

---

## Instructions for the AI

1. **Define the right MVP mindset**
   - Explain that minimum means smallest relative to the customer outcome,
     not smallest in feature count.
   - Explain that viable means the solution must actually deliver the desired outcome
     for the intended users in the intended context.
   - Prefer the term minimum viable solution for feature or capability-focused work.

2. **Warn against the bad MVP definition**
   - Call out the bad definition: the least usable or most painful product that still works.
   - Explain why that approach damages brand trust, user perception, and business value.
   - Emphasize that MVP is not an excuse for shipping a rough, barely usable product.

3. **Focus on outcomes, not output**
   - Recommend identifying specific outcomes and user behaviors before prioritizing work.
   - Explain that outcomes are the real goals behind the software and make prioritization possible.
   - Encourage teams to choose target outcomes for each release slice.

4. **Use story maps to slice releases**
   - Recommend using story maps to visualize what must be built for specific release outcomes.
   - Advise teams to slice maps left-to-right and move work into the first release only if it supports the target outcome.
   - Explain how this produces a release roadmap of outcome-focused slices.

5. **Treat the first MVP as a hypothesis**
   - Explain that any minimum viable solution is still a guess until validated.
   - Recommend designing the first release to test the riskiest assumptions.
   - Encourage smaller experiments or learning releases when uncertainty is high.

6. **Manage uncertainty and learning**
   - Ask teams to surface their biggest assumptions and risks explicitly.
   - Recommend building discovery and validation work into the plan, not only feature work.
   - Explain that the first MVP can be as much about learning as it is about shipping.

---

## Quality criteria

A strong MVP-focused response should make sure the team is:

- **outcome-driven:** success is defined by user impact,
- **realistic:** MVP is a hypothesis, not a guarantee,
- **disciplined:** minimality is judged against outcomes, not opinions,
- **risk-aware:** assumptions are visible and tested,
- **coherent:** the first release feels like a meaningful step, not a random subset.

---

## Example prompts

- "Help me decide what the smallest release is that will still achieve our user outcome."
- "How do we avoid building an MVP that customers would hate?"
- "What assumptions should our first release test if we want to learn quickly?"
