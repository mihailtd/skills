# Work Package Portfolio Map — Initiative-to-Strategy Execution Map

A Work Package Portfolio Map is a tabular architecture artifact that captures
work packages, initiatives, and the goals and objectives they support. It is a
foundational input for Architecture Roadmaps and execution planning.

This skill is for architects who need to translate strategy into discrete,
one-time work packages and verify that every strategic objective has an active
initiative assigned to it.

---

## When to use this skill

Use this skill when you need to:

- Document the initiatives, projects, and programs required to execute strategy.
- Link work packages directly to business goals and objectives.
- Identify strategic objectives without supporting initiatives.
- Prepare the inputs for a visual Architecture Roadmap.
- Support execution governance and progress tracking.

---

## Outcome

Produce a Work Package Portfolio Map that shows:

- **Goals and Objectives**: the strategic results the organization seeks.
- **Work Packages / Initiatives**: the one-time activities, projects, or programs
  designed to deliver those results.
- **Owner**: the business actor, team, or function accountable for the work
  package.
- **Status / attributes**: optional fields like cost, progress, or priority.

The map should clearly show which objectives are covered by initiatives and where
strategy execution gaps remain.

---

## Step-by-step workflow

1. **Collect strategy artifacts**
   - Start with the documented goals, objectives, and intended results from the
     strategy and planning phase.
   - Use existing deliverables such as goal/objective matrices and strategy maps.

2. **Define work packages**
   - Identify initiatives, projects, programs, or work packages that will execute
     the strategy.
   - Treat initiatives and work packages as equivalent for the purpose of this
     map.

3. **Map work to objectives**
   - Cross-reference each work package against the goal(s) or objective(s) it
     supports.
   - Record the accountable owner and any execution attributes that matter.

4. **Build the tabular map**
   - Create a table with columns such as Goal / Objective, Work Package,
     Owner, and Status / Notes.
   - Capture the relationship between strategy and execution in each row.

5. **Identify gaps and overlaps**
   - Highlight objectives with no assigned work packages.
   - Flag duplicate or overlapping initiatives that support the same objectives.

6. **Use the map to create a roadmap**
   - Use the table as the source for a timeline-based Architecture Roadmap.
   - Plot goals and objectives on the vertical axis and work packages across the
     horizontal timeline.

7. **Review and maintain**
   - Validate the map with strategy sponsors, program owners, and delivery teams.
   - Keep it updated as initiatives change, are completed, or are re-prioritized.

---

## Decision points and guidance

- **Work package definition**: include only one-time initiatives, not repeatable
  business-as-usual processes.
- **Map format**: prefer a tabular structure because the deliverable is a map,
  not a diagram.
- **Owner and governance**: capture clear accountability for each work package.
- **Strategic coverage**: use the map to expose missing initiative coverage or
  excessive overlap.
- **Roadmap readiness**: ensure the table contains enough timeline and status
  detail to feed the Architecture Roadmap.

---

## Quality criteria

A strong Work Package Portfolio Map should be:

- **Strategically linked**: each work package is tied to a goal or objective.
- **Complete**: captures all major initiatives required to execute the strategy.
- **Gap-aware**: identifies objectives without initiatives.
- **Actionable**: includes owners, status, or other attributes for execution.
- **Roadmap-ready**: usable as the basis for a timeline view of strategy
  implementation.
