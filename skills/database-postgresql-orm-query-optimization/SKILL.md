# PostgreSQL ORM Query Optimization

This skill helps developers optimize poorly generated ORM queries by addressing
common ORM pitfalls, eliminating N+1 behavior, and applying a NORM (No-ORM)
architecture with JSON-based database contracts.

---

## 1. Identify the Core ORM Pitfalls and the N+1 Problem

- **Object-Relational Impedance Mismatch:** ORMs often fail to translate complex
  object graphs into efficient set-based SQL, creating an impedance mismatch
  between application objects and relational data.
- **The N+1 problem:** This occurs when an ORM loads a parent object with one
  query and then issues an additional query for each child record, resulting in
  many small, inefficient database calls.
- **The shopping list anti-pattern:** ORMs may break objects into atomic pieces
  and execute many separate queries, which behaves like a slow "shopping list"
  of database calls.

---

## 2. Introduce the NORM (No-ORM) Architecture

- **The alternative approach:** NORM avoids ORM-generated SQL and treats the
  database as a functional data boundary.
- **Single database calls:** The goal is to persist or retrieve a complex,
  nested object with one database call, regardless of its internal structure.
- **Reduced roundtrips:** This architecture minimizes network latency and
  eliminates the N+1 pattern by keeping the object transformation logic in the
  database.

---

## 3. Establish JSON-Based Contracts

- **Data serialization:** Serialize complex application objects into JSON
  payloads or arrays of JSON text before sending them to the database.
- **The contract:** The JSON payload becomes a strict transfer contract between
  application code and database functions.
- **Independent development:** Application and database developers can work in
  parallel as long as they agree on the JSON schema.

---

## 4. Utilize Database Functions for Data Exchange

- **Processing in the database:** Create PostgreSQL functions that accept JSON
  contracts, parse them, and execute the required logic internally.
- **Execution flow:** The application passes a JSON payload to the function,
  which interprets it and performs searches, transformations, or DML.
- **Return payloads:** Have the function return JSON objects or text strings so
  the application can deserialize the complete result in one step.

Example:

```sql
CREATE FUNCTION get_order_details(order_json jsonb)
RETURNS jsonb LANGUAGE plpgsql AS $$
BEGIN
  -- parse order_json, query underlying tables, build and return JSON
  RETURN jsonb_build_object('order', order_json);
END;
$$;
```

---

## 5. Optimizing Updates and Data Manipulation (DML)

- **Complex updates:** Send the entire update request as a single JSON object.
- **Internal distribution:** The database function parses the JSON and executes
  the necessary `INSERT`, `UPDATE`, or `DELETE` operations across multiple
  tables within a single transaction.
- **Bypassing ORM slowness:** This avoids ORM row-by-row updates and keeps DML
  in the database where set-based operations perform best.

---
